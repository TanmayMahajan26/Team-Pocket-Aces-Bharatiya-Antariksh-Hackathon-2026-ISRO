# ISRO PS14 Space Weather: Frontend Developer Context

Welcome to the project! This document outlines everything you need to know to build a state-of-the-art frontend dashboard for the ISRO Geostationary Electron Flux Forecasting System.

The backend is built with **FastAPI**, **PyTorch** (Time-Series Transformer), and **TimescaleDB** (PostgreSQL). It runs a background daemon that polls NOAA REST APIs every 5 minutes and runs inference to predict dangerous Space Weather anomalies up to 12 hours in advance.

---

## 🚀 The Mission (What You're Building)
You need to build a modern, high-tech dashboard (think NASA/SpaceX control room aesthetic: dark mode, glowing accents, real-time telemetry). The dashboard needs to clearly communicate:
1. The **current** >2 MeV electron flux (is it dangerous right now?).
2. The **predicted** flux for 30-min, 6-hour, and 12-hour horizons.
3. The **uncertainty bands** (95% worst-case scenarios).
4. **Why** an alert is happening (Feature Explainability).
5. **System Health** (Is the backend actually connected to NOAA?).

---

## 📡 API Endpoints

The FastAPI server runs locally on `http://localhost:8000`. You can hit these endpoints directly using `fetch` or `axios`. CORS is fully enabled for all origins `["*"]`.

### 1. The Real-Time Feed
**`GET /api/live`**
This is your main endpoint. It returns the absolute latest telemetry and forecast generated by the PyTorch model.
*Recommendation: Poll this endpoint every 30 seconds.*

**Response Payload:**
```json
{
  "timestamp": "2026-06-28T12:00:00+00:00",           // Time the prediction was made
  "latest_data_time": "2026-06-28T11:55:00+00:00",    // The timestamp of the actual NOAA data used
  "connection_status": "CONNECTED",                   // "CONNECTED" or "DISCONNECTED"
  "current_flux_gt2MeV": 2.5,                         // Current satellite reading
  "status": "Normal",                                 // "Normal", "WARNING", or "CRITICAL"
  "predictions_pfu": [5.6, 8.1, 15.2],                // AI Predictions for [30m, 6h, 12h]
  "p95_pfu": [8.5, 12.0, 28.1],                       // 95% worst-case upper bound for [30m, 6h, 12h]
  "explainability": {
    "top_drivers": {
      "Bz_GSM": 45.2,                                 // % importance of this physical parameter
      "Vsw": 22.1,
      "ULF_proxy": 15.4
    },
    "method": "Input Gradients (Saliency) on 30-min Forecast"
  }
}
```

### 2. Historical Charting
**`GET /api/history?limit=100`**
Returns an array of the `ForecastData` objects (same schema as above). 
*Recommendation: Call this once on page load to render a Time-Series Line Chart (e.g., using Recharts or Chart.js) showing how the predictions drift over time against the actual current flux.*

## 🎨 UI/UX Requirements & Edge Cases to Handle

1. **The "Status" Badge**:
   - `Normal` (< 100 pfu): Render as Green.
   - `WARNING` (100 - 999 pfu): Render as Yellow/Orange. Pulse animation.
   - `CRITICAL` (>= 1000 pfu): Render as Red. This is an extreme storm. Flashing red border on the screen.
2. **Network Resilience Warning**:
   - If `connection_status === "DISCONNECTED"`, display a red banner stating **"STALE DATA: NOAA API Offline."** The system is built to safely retry in the background, but the scientists *must* know if the dashboard is lagging behind real-time.
3. **The Explainability Module**:
   - Scientists hate "black-box" AI. In your UI, create a panel called "Primary Storm Drivers".
   - Iterate over `explainability.top_drivers` and render progress bars representing the percentage. E.g., `Bz_GSM [======    ] 45.2%`.
4. **Charting Uncertainty**:
   - When plotting the line chart, plot `predictions_pfu` as a solid line, and plot `p95_pfu` as a shaded upper-bound area. This represents the AI's 95% confidence interval for worst-case scenarios.

## 🛠 How to run the Backend Locally

If you don't have Docker installed, just open two terminals in the root folder:

**Terminal 1 (The Polling Daemon):**
```bash
python run_live_service.py
```
*(This triggers the AI to fetch data and write to the local SQLite database).*

**Terminal 2 (The Web Server):**
```bash
uvicorn api:app --host 0.0.0.0 --port 8000
```
*(This exposes the API to your frontend).*
